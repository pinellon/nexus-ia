# Nexus AI v0.3 Controlled Autonomy Plan

Status: planning only. No v0.3 autonomy is implemented in this PR.

## Goal

v0.3 should introduce controlled autonomy for local development tasks while keeping human approval as the boundary for every write, patch application, command with side effects, rollback, dependency change, and long-running operation.

The target is not a fully autonomous coding model. The target is a supervised task runner that can decompose work, queue steps, ask for approval, execute allowed actions, log decisions, and stop safely.

## Non-Goals

- No model training.
- No custom model release.
- No tokenizer work.
- No cloud training pipeline.
- No product or sales packaging.
- No automatic patch application.
- No broad UI redesign.
- No uncontrolled shell access.

## Task Queue

The v0.3 runner should maintain an explicit task queue with:

- queue id
- task id
- parent task id when spawned from another step
- requested mode
- user prompt
- current status: `queued`, `planning`, `waiting_for_approval`, `running`, `cancelled`, `failed`, `completed`, `rolled_back`
- created/updated timestamps
- execution budget
- approvals required
- artifacts produced
- audit log references

The queue should support one active task at a time by default. Parallel execution can be planned later, but v0.3 should avoid parallel writes.

## Multi-Step Planner

The planner should produce a reviewable plan before any action runs.

Each step should include:

- step id
- title
- purpose
- expected input
- expected output
- files or directories touched
- command intent if a command is needed
- risk level
- required approval type
- rollback expectation

The planner should classify steps as:

- `read_only`
- `analysis`
- `test`
- `patch_proposal`
- `write_request`
- `command_request`
- `rollback_request`

Only `read_only`, `analysis`, and safe `test` steps may run without explicit per-action approval, and only if they stay inside the workspace and match allowlisted commands.

## Human Approval Boundary

Human approval is required before:

- applying a patch
- writing or deleting files
- moving files
- installing dependencies
- running non-allowlisted commands
- running commands with network effects
- changing environment/config/secrets
- triggering rollback
- exceeding the task budget

Approval payloads should show:

- action type
- exact command or patch
- target files
- risk assessment
- expected result
- rollback plan

Approval should be explicit and scoped to one action or a small batch of related actions.

## Patch Policy

Patch generation and patch application must stay separate.

v0.3 may prepare patches, diffs, and staged artifacts, but:

- `auto_applied` remains `false`
- patch application requires approval
- stale-file checks must run before application
- patch previews must be readable before approval
- patch metadata must include source task and planner step

## Rollback

Rollback should be real, auditable, and tested.

Rollback should support:

- restoring files changed by an approved patch
- verifying expected before/after hashes
- recording rollback reason
- exposing rollback status in the task queue
- detecting rollback failure honestly

Rollback should not claim success unless the file state is restored and verified.

## Sandbox

The sandbox should enforce:

- workspace boundary checks
- command allowlist
- blocked dangerous command patterns
- timeout per command
- stdout/stderr caps
- no shell by default
- no recursive destructive operations
- no writes outside the repository

Commands should run with argument arrays where possible. Shell execution should remain blocked unless a future approved design adds a narrowly scoped exception.

## Permissions By Action

Permissions should be represented as structured capabilities:

- `read_files`
- `list_files`
- `run_tests`
- `propose_patch`
- `apply_patch`
- `write_file`
- `delete_file`
- `install_dependency`
- `network_request`
- `rollback`

Each task should declare required capabilities during planning. The runner should reject actions not present in the approved capability set.

## Logs And Audit

Every task should write an audit trail with:

- queue id
- step id
- action id
- actor: user, planner, bridge, runner
- input summary
- output summary
- exact command args when applicable
- changed files
- approval decision
- timestamps
- duration
- errors
- rollback references

Logs should be structured JSON for machines and summarized in Markdown for humans.

## Cancellation

v0.3 should support cancellation at:

- queued task level
- running task level
- child process level
- waiting-for-approval level

Cancellation should:

- kill running child processes
- mark the task as `cancelled`
- preserve logs
- never apply pending patches
- offer rollback only for already approved/applied changes

## Execution Limits

Default limits:

- max active tasks: 1
- max task duration: 10 minutes
- max command duration: 2 minutes
- max planner steps: 12
- max files in one patch: 10
- max stdout/stderr per action: 1 MB
- max retries per step: 1

Any budget increase should require human approval.

## Required Tests

Unit tests:

- planner emits structured multi-step plans
- queue status transitions are valid
- unsupported capabilities are rejected
- approval is required before writes
- `auto_applied` remains false
- sandbox blocks dangerous commands
- root traversal is rejected
- timeout cancels child process
- rollback verifies restored content
- cancellation preserves audit records

API tests:

- create task
- inspect queue
- approve action
- reject action
- cancel task
- retrieve audit log
- rollback approved change

Integration tests:

- read-only task completes without write approval
- patch proposal waits for approval
- approved patch applies through existing patch review path
- rejected patch leaves files untouched
- cancelled command stops and records status

Smoke tests:

- controlled autonomy smoke suite with no auto-apply
- rollback smoke with verified restoration
- dangerous command smoke

## Acceptance Criteria

v0.3 can be considered ready only when:

- v0.2 remains stable on `main`
- task queue is explicit and inspectable
- planner output is structured and reviewable
- every write requires approval
- `auto_applied` is still false
- sandbox blocks dangerous commands
- rollback is verified by tests and smoke
- cancellation works for running tasks
- logs/audit records are available
- gates pass:
  - `npm run typecheck`
  - `npm run build`
  - `npm test`
  - Python compile for `NexusAI/*.py`
  - controlled Python unit tests
  - coder component tests
  - v0.3 autonomy smoke suite

## Proposed Implementation Order

1. Define task queue and status model.
2. Add planner response schema and validation.
3. Add approval action schema.
4. Wire sandboxed command execution to queue steps.
5. Wire patch proposal to existing patch review flow.
6. Add cancellation for active child processes.
7. Add rollback verification path.
8. Add audit log persistence.
9. Add API endpoints for queue, approvals, cancellation, and audit.
10. Add minimal UI for queue and approvals.
11. Add tests and smoke suite.

## Explicit Hold

Do not implement this plan in the planning PR. The next PR should only contain this document so v0.3 can be reviewed before code begins.
