# NexusAI v0.3 – Controlled Autonomy: Task Queue & Approval Gates

## Overview

This document describes the first step of NexusAI v0.3: a **controlled-autonomy layer** that introduces a task queue, multi-step plans, mandatory human approval gates, an audit log, and CLI commands for managing the lifecycle of a task.

**Key safety guarantee:** No mutable action (patch, file write, command) is ever applied automatically. `auto_applied` remains `false` in every API response.

---

## Objectives

- Allow NexusAI to plan multi-step tasks without executing them automatically.
- Gate every mutable action behind explicit human approval.
- Record all decisions and lifecycle events in a tamper-evident audit log.
- Provide a simple CLI workflow for plan → review → approve/reject → cancel.

---

## Limits (v0.3 Step 1)

| What we DON'T do | Why |
|---|---|
| Auto-apply patches | Safety: human must review every mutation |
| Execute mutable commands | Safety: ditto |
| Train a model | Out of scope for this step |
| Start a product / sell anything | Out of scope for this step |
| Modify v0.2 UI | Separate concern |

## Task Status States

```
pending → planned → waiting_approval → approved → running → completed
                                  ↘ cancelled ↙
                             failed ← running
```

| Status | Meaning |
|---|---|
| `pending` | Task created, plan not yet generated |
| `planned` | Plan generated, no mutable steps pending |
| `waiting_approval` | One or more mutable steps need human review |
| `approved` | All required approvals have been recorded (step-level) |
| `running` | Reserved for future auto-run gating |
| `completed` | Task finished successfully |
| `failed` | Task failed |
| `cancelled` | Task cancelled by user |

---

## Step Action Types

| Type | Mutable | Requires Approval |
|---|---|---|
| `read_only` | No | No |
| `analysis` | No | No |
| `test` | No | No |
| `patch_proposal` | **Yes** | **Yes (auto)** |
| `write_request` | **Yes** | **Yes (auto)** |
| `command_request` | **Yes** | **Yes (auto)** |
| `rollback_request` | **Yes** | **Yes (auto)** |

**Rule:** If `mutable = true`, then `requires_approval = true` is automatically enforced, regardless of what the caller passes.

## CLI Commands

All commands output **valid JSON** to stdout. All outputs include `"auto_applied": false`.

### `autonomy-plan`

Creates a task and generates a default multi-step plan. Returns the `task_id` and list of steps with their `step_id`s. Mutable steps are immediately placed in `waiting_approval`.

```bash
python NexusAI/repo_mode.py autonomy-plan --task "add input validation" --root .
```

Example output:
```json
{
  "task_id": "3f4a8b21-...",
  "status": "waiting_approval",
  "prompt": "add input validation",
  "steps": [
    {"step_id": "uuid-1", "description": "Analyse codebase", "action_type": "read_only", "mutable": false, "status": "pending"},
    {"step_id": "uuid-2", "description": "Propose patch", "action_type": "patch_proposal", "mutable": true, "status": "waiting_approval", "is_approved": false}
  ],
  "auto_applied": false
}
```

### `autonomy-status`

Returns current task status and per-step approval decisions.

```bash
python NexusAI/repo_mode.py autonomy-status --task-id "<TASK_ID>"
```

### `autonomy-approve`

Records an approval for a specific step. **Does NOT execute the step.**

```bash
python NexusAI/repo_mode.py autonomy-approve \
  --task-id "<TASK_ID>" \
  --step-id "<STEP_ID>" \
  --reason "Reviewed and approved"
```

### `autonomy-reject`

Records a rejection. The step is blocked.

```bash
python NexusAI/repo_mode.py autonomy-reject \
  --task-id "<TASK_ID>" \
  --step-id "<STEP_ID>" \
  --reason "Too risky"
```

### `autonomy-request-changes`

Records a request for changes on a step. Step remains in `waiting_approval`.

```bash
python NexusAI/repo_mode.py autonomy-request-changes \
  --task-id "<TASK_ID>" \
  --step-id "<STEP_ID>" \
  --reason "Please adjust the scope"
```

### `autonomy-cancel`

Cancels the entire task and all pending steps.

```bash
python NexusAI/repo_mode.py autonomy-cancel \
  --task-id "<TASK_ID>" \
  --reason "Decided not to proceed"
```

## Approval Flow

```
User: autonomy-plan --task "..."
  → Task created (UUID v4)
  → Plan generated (multi-step)
  → Mutable steps → status: waiting_approval
  → approval_required events logged to audit log

User: autonomy-status --task-id <ID>
  → See all steps and their current approval state

User: autonomy-approve --task-id <ID> --step-id <SID>
  → Decision recorded in ApprovalGate
  → Step status: approved
  → approval_granted event logged
  → auto_applied: false (NOTHING IS EXECUTED)

User: autonomy-reject --task-id <ID> --step-id <SID>
  → Decision recorded
  → Step status: cancelled
  → approval_rejected event logged
```

---

## Audit Log

Events are written to `NexusAI/memory/audit_log.jsonl` in newline-delimited JSON format.

### Event Types

| Event | Trigger |
|---|---|
| `task_created` | `autonomy-plan` |
| `plan_created` | Plan generated |
| `approval_required` | Each mutable step in the plan |
| `approval_granted` | `autonomy-approve` |
| `approval_rejected` | `autonomy-reject` |
| `changes_requested` | `autonomy-request-changes` |
| `task_cancelled` | `autonomy-cancel` |
| `task_completed` | Future |
| `task_failed` | Future |

### Secret Sanitisation

Values for the following keys are **always redacted** to `"****"` before writing:

`token`, `secret`, `password`, `api_key`, `private_key`, `credential`,
`authorization`, `access_token`, `refresh_token`

Future hardening: regex-based masking of secrets embedded in longer strings.

### Example Entry

```json
{"event_type": "approval_granted", "timestamp": "2026-06-08T19:45:00+00:00", "task_id": "3f4a8b21-...", "step_id": "uuid-2", "details": {"reason": "Reviewed and approved", "auto_applied": false}}
```

---

## Why Auto-Apply Stays Disabled

Auto-apply would allow NexusAI to modify the codebase without human review, which is a significant safety risk for:
- Unintended data loss or corruption
- Accidental exposure of secrets
- Breaking production code

Until a **rollback/sandbox** layer (v0.3.2) is in place that can safely undo mutations, auto-apply will remain off.
