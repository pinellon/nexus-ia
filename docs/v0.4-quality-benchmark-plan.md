# Nexus AI v0.4 Quality Benchmark and Operational Hardening Plan

## Scope

v0.4 starts after v0.3.3 is merged into `main`. The goal is to measure and harden the local controlled programming workflow:

`UI -> plan -> approve -> execute controlled step -> rollback -> audit log`

This phase does not start model training, tokenizer work, cloud training, billing, login, product sales work, or auto-apply.

## Benchmark Fixture

The canonical v0.4 benchmark fixture is:

- `benchmarks/coder_smoke_100.json`

It contains 100 measurable coder tasks across these categories:

- repo indexing
- multi-file planning
- simple bugfix
- test creation
- docs changes
- small refactor
- security review
- rollback
- human approval
- UI autonomy flow
- API endpoint coverage
- persistence/restart recovery
- operational hardening

Each case includes:

- `id`
- `category`
- `mode`
- `task`
- `expected_checks`

The fixture is intentionally static and deterministic. It is a benchmark definition, not a training dataset.

## Metrics To Report

Every v0.4 benchmark report should include:

- `success_rate`
- `fallback_usage_rate`
- `model_direct_pass_rate`
- `repair_success_rate`
- `rollback_success_rate`
- `average_task_time`
- `failed_by_category`
- `task_category_summary`

If a metric cannot be computed from the current runner output, the report must mark it as `not_measured` instead of guessing.

## Operational Hardening Checks

v0.4 must keep these checks visible:

- no tracked zip/handoff/cache/log artifacts
- ignored local zip artifacts stay out of git
- `reports/`, coverage, and runtime caches are ignored
- desktop packaging remains documented as a placeholder
- `NexusAI/memory/*.json` and `NexusAI/memory/*.jsonl` stay runtime state, not source
- `/api/nexus/health` reports `auto_apply_enabled:false`
- `/api/nexus/run` rejects invalid bodies, unsupported modes, and root traversal
- `/api/nexus/autonomy/*` preserves `auto_applied:false`
- `/repo/autonomy/execute` requires `approved:true` and exactly one action payload
- Windows and Linux path handling reject traversal and null bytes

## Persistence Notes

The current persistence layer remains JSON based:

- `NexusAI/memory/task_queue.json`
- `NexusAI/memory/autonomy_plans.json`
- `NexusAI/memory/audit_log.jsonl`

This is acceptable for v0.4 local use. It supports restart recovery for task queue, plan status, and audit history. Known limitations:

- no concurrent writer coordination beyond local process discipline
- no transactional database semantics
- corrupted JSON starts fresh or requires manual recovery depending on the component
- no query/index performance guarantees for large histories

SQLite or another stronger store is deferred to the persistence phase after quality measurement.

## Manual Validation

Before merging a v0.4 hardening PR:

1. Open the local UI.
2. Create an autonomy task.
3. Approve a mutable step.
4. Execute the approved step.
5. Roll back the change.
6. Confirm audit log events are visible.
7. Confirm every response still reports `auto_applied:false`.

## Non-Goals

- No model training.
- No custom model work.
- No tokenizer/cloud training pipeline.
- No auto-apply.
- No billing/login.
- No large UI refactor.
- No `package.json` or `package-lock.json` changes unless a test proves they are necessary.
