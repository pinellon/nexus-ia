O NexusAI tem um pipeline acima da media para um projeto local de IA: dataset auditavel, logs, checkpoints,
benchmark e inferencia. O gargalo principal nao e organizacao do projeto; e capacidade do modelo, separacao de

4. Como melhorar o dataset
O dataset atual gold de 300 pares e limpo, mas pequeno e heterogeneo. A distribuicao tem Python, HTML, TSX,
text/debug, TypeScript, diff e JSON. Para um modelo pequeno, essa mistura dilui o aprendizado. A melhoria mais
importante e transformar o dataset em trilhas separadas por dominio, com validacao automatica antes de entrar no
treino.
Problema Impacto Solucao pratica
Poucos exemplos gold O modelo memoriza estilo mas nao
generaliza.
Criar 2k-5k exemplos Python/Flask
validados automaticamente.
Mistura de dominios Prompt de CPF vira Flask; prompt React
vira JSON.
Separar datasets e usar tags de dominio
no prompt.
Sem garantia de compilacao real compile_like_rate fica baixo. Rodar parser/teste em todo exemplo
antes de aceitar.
Respostas com markdown variavel Extractor confunde ou modelo gera lixo
antes do codigo.
Padronizar: codigo puro ou bloco unico,
nunca ambos no mesmo split.
Prompts genericos Modelo nao aprende restricoes precisas. Prompts com contrato: linguagem,
entrada, saida, regras e formato.
4.1 Formato recomendado de exemplo SFT
Cada exemplo deve conter contrato claro. O modelo precisa saber linguagem, dominio, restricoes e formato
esperado. Isso reduz mistura de contextos.
### Domain: python_flask
### Task: generate_code
### Constraints:
- Responda apenas com codigo Python valido.
- Use Flask, request e jsonify.
- Crie rota POST /api/v1/tasks.
- Se title estiver vazio, retorne erro 400.
### Instruction:
Crie uma API Flask com rota POST /api/v1/tasks, validacao JSON e resposta de erro quando o titulo estiver vazio.
### Response:
from flask import Flask, jsonify, request
app = Flask(__name__)
tasks = []
@app.post("/api/v1/tasks")
def create_task():
    data = request.get_json(silent=True) or {}
    title = str(data.get("title", "")).strip()
    if not title:
        return jsonify({"error": "title is required"}), 400
    task = {"id": len(tasks) + 1, "title": title, "done": False}
    tasks.append(task)
    return jsonify(task), 201
4.2 Regras de entrada no dataset
 Todo exemplo Python precisa passar em ast.parse antes de entrar no treino.
 Todo exemplo Flask precisa passar em importacao simples sem executar servidor.
 Todo JSON precisa passar em json.loads.
 Todo TSX precisa ficar fora do dataset principal ate Python/Flask estabilizar.
 Todo exemplo deve ter uma tag de dominio e uma tag de tarefa.
 Deduplicar exemplos por hash normalizado do prompt e da resposta.
 Separar train/validation/test antes de gerar metricas, com test congelado.

5.3 Como interpretar resultados
 Se loss melhora e benchmark piora, o modelo esta memorizando padroes de treino.
 Se ast_pass sobe mas instruction_following fica baixo, o modelo escreve codigo valido para a tarefa errada.
 Se repetition_score sobe mas compile cai, a decodificacao ficou menos repetitiva mas ainda sem estrutura.
 Se repair_success sobe, o produto pode ficar utilizavel mesmo antes do modelo base ficar perfeito.
 Escolha checkpoint por score de benchmark, nao apenas por loss de treino.

Conclusao
O NexusAI nao deve ser julgado apenas pelo checkpoint atual. O modelo atual ainda nao e bom como coding
assistant, mas a engenharia do projeto mostra maturidade: ha logs, benchmark, auditoria, metricas e objetivo de
produto. O proximo salto vem de uma mudanca de estrategia: menos full-stack misturado, mais curriculo por
dominio, mais validacao real, mais repair loop e mais produto ao redor do modelo.
A direcao recomendada e clara: congelar o baseline 9,4M, migrar para v2/v3, separar Python/Flask de React,
aumentar dataset validado, medir por compilacao real e construir o NexusAI como assistente local com patch review,
memoria e preview. Assim, mesmo antes de atingir qualidade de modelos grandes, o sistema pode virar uma
ferramenta real para acelerar o desenvolvimento do proprio Nexus.
